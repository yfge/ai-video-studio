# Media Asset Persistence (OSS/CDN)

This doc describes how generated media (image/video/audio) is persisted to OSS/CDN, how object keys are named, and which metadata is attached.

## Where The Code Lives

- Upload entrypoints: `ai-pic-backend/app/services/media/media_persistence.py`
  - `upload_from_url()`: download remote URL then upload to OSS
  - `upload_bytes()`: upload bytes to OSS
  - `upload_base64()`: decode base64 then upload to OSS (adds `sha256`)
- OSS SDK integration: `ai-pic-backend/app/services/storage/oss_service.py` and `ai-pic-backend/app/services/storage/oss_upload_mixin.py`

## OSS Object Key Naming

OSS keys are generated by `OSSUploadMixin._generate_object_key()` and follow:

```
{prefix}/{file_type}/{YYYYMMDD}/{HHMMSS}/{uuid8}{ext}
```

Examples (current defaults used by generation pipelines):

- Video: `ai-generated/videos/video/20260129/094426/e3cb185e.mp4`
- Audio: `ai-generated/audio/audio/20260129/100516/fc33d123.mp3`
- Thumbnails: `ai-generated/thumbnails/image/20260129/094426/....png`
- Video last frame: `ai-generated/video-last-frames/image/20260129/094426/....png`

Notes:

- `prefix` is chosen by the caller (see `media_persistence.py` defaults).
- `file_type` is one of `image|video|audio` and is always inserted as a path segment.
- `ext` is inferred from filename (and defaults for some types).

## OSS Metadata Contract (x-oss-meta-\*)

We attach generation metadata as OSS object metadata headers (`x-oss-meta-*`).

Important constraints:

- Keys and values must be ASCII; non-ASCII values are dropped (OSS header safety).
- Values are coerced to strings.

Standard metadata keys (built by `build_generation_metadata()`):

- `media_type`: `image|video|audio`
- `provider`: e.g. `google`
- `model`: e.g. `veo-3.1-generate-preview`
- `mime_type`: optional (best-effort)
- `duration_seconds`: optional
- `fps`: optional
- `resolution`: optional (e.g. `720p`)
- `sha256`: optional (auto-populated by `upload_base64()`; empty when unknown)
- `generated_at`: UTC ISO timestamp

Callsites can add extra ASCII-only keys (examples):

- Audio: `voice_type`, `speed`, `text` (truncated), `text_len`
- Video: `end_image_url` (optional; may be long)
- Images: caller-specific trace fields (e.g. `virtual_ip_id`, `environment_id`)

## Where Rich (Non-ASCII) Data Should Live

Prompts and other rich data often contain non-ASCII (e.g. zh-CN). Those should not rely on OSS object metadata.

Instead:

- Store prompts and parameters in DB (`Task.parameters`, `VideoGenerationTask.prompt/parameters/result`, etc.).
- Persist audit trails to `Task.parameters.agent_run` where applicable.

## DB Generation Metadata (Video Tasks)

Storyboard video generation persists normalized metadata to
`video_generation_tasks.generation_metadata` (JSON) so consumers do not need to
parse provider-specific payloads from `result`.

Current fields (best-effort):

- `provider`, `model`, `task_id`/`provider_task_id`, `model_type`
- `duration_seconds`, `fps`, `resolution`, `ratio`, `width`, `height`
- `assets.video|thumbnail|last_frame` (URL/object_key/file_size/mime_type/sha256)

## Current `media_assets` Transition Contract

Timeline Spec v1 now has a unified DB asset row for image, video, audio, and
subtitle outputs. The `media_assets` table does not replace current OSS
metadata, `images`, or `video_generation_tasks` in one step; it gives timeline
clips, replacement actions, and render jobs one stable id space.

Current fields:

- `id`
- `asset_type`: `image` | `video` | `audio` | `subtitle`
- `origin`: current values include `ai_generated`, `upload`, `imported`,
  `rendered`, `timeline_spec`, `legacy`, and `provider_rework`
- `file_url`
- `object_key`
- `file_path`
- `mime_type`
- `duration_ms`
- `width`
- `height`
- `hash`
- `metadata`
- `created_by`
- `created_at`

Timeline references:

- Dialogue clips use `audio_asset_id` or `asset_ref.audio_asset_id` when an
  asset id is already available, and may still carry URL/object locators during
  transition.
- Video clips use `start_frame_asset_id`, `end_frame_asset_id`,
  `video_asset_id`, or explicit `*_asset_ref` objects for start frame, end
  frame, storyboard image, storyboard video, and generated video assets.
- Subtitle clips use `subtitle_asset_id` only when a standalone subtitle file is
  emitted; inline subtitle text may remain in the clip spec for editing.
- Render jobs write `output_asset_id` for proxy/final exports.

Timeline clip lineage:

- `timeline_clip_assets` links `timeline_id`, `timeline_version`, stable
  `clip_id`, `asset_role`, and `media_asset_id`.
- Current asset roles include `source_audio`, `start_frame`, `end_frame`,
  `storyboard_image`, `storyboard_video`, `generated_video`, and
  `render_output`.
- Operator rework actions add replacement lineage through the same stable
  `clip_id`; `replacement_of_id` points at the previous active asset for that
  role when one exists.
- Provider-backed video rework writes successful outputs with
  `origin=provider_rework`, records `provider_rework` lineage, and can trigger a
  final render with a rework fingerprint in the preset.

Transition rules:

- Existing `scripts.extra_metadata.storyboard.frames[*].image_url` and
  `video_url` remain compatibility fields until migrated.
- Existing `video_generation_tasks` remains the provider polling/audit record.
  Its normalized `generation_metadata.assets.*` fields should map to or create
  `media_assets` rows when the Timeline path needs stable clip lineage.
- OSS object metadata remains ASCII-only and lightweight. Rich prompts,
  non-ASCII text, provider payloads, and timeline source references belong in
  DB JSON fields such as `Task.parameters`, `video_generation_tasks`, timeline
  clips, or `media_assets.metadata`.

## Expected Flow (Storyboard Video)

For storyboard video generation:

1. Task is created (`Task`).
2. Provider task is submitted and stored (`video_generation_tasks`).
3. Poller detects completion, uploads media to OSS/CDN, and writes the final URLs into:
   - `video_generation_tasks.result` (JSON string)
   - `scripts.extra_metadata.storyboard.frames[*].video_url` (for the relevant frame)
4. Timeline implementation additionally creates or links `media_assets` rows,
   records `timeline_clip_assets` lineage, and references those ids from
   Timeline Spec v1 clips or render jobs where available.
