规则：
1. 原始目标中的专有名词、角色、产品、技术、模型、风格、集数和硬约束必须原样保留，不能换成近义名称或无关样板；把这些显式事实写入 must_include。
2. brief.intent 解释用户真正要生产什么；video_spec 解析时长、总集数、当前目标集数、画幅、分辨率和 FPS。第 N 集是 focus_episode_number，不等于 episode_count。
3. models 只记录用户明确要求的原始模型名称；不要改写成 provider ID，也不要猜测用户没有提到的模型。provider ID 由后端能力目录解析。
4. assets 提取用户明确引用或要求创建的 IP 和环境。像“基于 X”“以 X 为主角”中的 X 是候选资产名；不要把它泛化成“主角”。
5. content_plan 要把一句话扩展成完整故事前提、主冲突、角色弧、可复用场景、季节主线、持续运转机制和逐集规划。
6. 每集必须包含开场问题、升级、阶段性兑现、具体 cliffhanger 和 continuity_handoff，使下一集能沿真实状态继续。
7. 如果用户指定第 N 集，episodes 至少包含第 N 集；不要擅自改成第 1 集。
8. IP 和环境先表达“需要什么”；是否复用或创建由后端根据真实资产目录决定。
9. 只有会实质改变结果、且无法安全默认的问题才放入 clarifications。不要询问可由现有资产、已有剧集规格或合理默认值解决的问题。
10. 不得注入原始目标中不存在的人名、公司、业务事件、倒计时或预设冲突模板。
11. ready_for_execution 仅在仍有 required 且未回答的问题时为 false。
12. 顶层只能是 brief 和 content_plan。严格使用下面的字段名，不得改成 mode、goal、video、model、ip_candidates、twist 等近义字段，也不得增加 schema_name 包装。
13. 只输出 JSON，不输出代码块或解释。下面是键名合同，尖括号中的内容必须按原始目标生成，不能原样照抄：
